# Overview & Feature Inventory

This document was created with the assistance of an AI.

## One-sentence pitch
A glitch-mythic colony-sim where you alternate between **Champion** (grounded actor) and **Deity** (rule-bending influencer) across a repeating **day/night loop**, steering a settlement caught between **Pantheon politics** and the **Gates of Aker**—a boundary where **Primordial Chaos** leaks into reality as black butterflies.

## Player fantasy
- “I build and protect a living settlement.”
- “I become legend, then become the force that shapes legend.”
- “I perform miracles that have consequences and must be believed.”
- “I navigate pantheon social dynamics and cosmic glitches.”

## High-level structure
- **Day (Champion Phase):** direct action, limited power, embodied risk.
- **Night (Deity Phase):** indirect action, miracles, influence, cosmology.
- Persistent **World State** carries between phases (and between lives).

## Core feature inventory (as discussed)
### Role loop
- Champion ↔ Deity alternation (day/night)
- Role-specific verbs, constraints, and UI

### Pantheon & miracles
- Pantheon of “glitch gods” with agendas and rivalries
- Miracle system: costs, cooldowns, public interpretation, and verification
- “Miracle verification” / trust economy: the world decides what was real

### Colony simulation
- Settlement construction, production chains, and resource management
- Population agents with needs, jobs, relationships
- Threats (rifts, invasions, “tide of battle” moments)

### Lore & tone
- Mythic Greek structure remixed as sci‑fi glitch cosmology
- Prometheus / Pandora / Epimetheus motifs
- Black butterflies as Chaos shards
- “Duck people in space future ancient glitch greece” aesthetic thread
- Recurring “Promethean error loop” motif (world/system failing musically)

### Progression
- Champion retirement / death → legacy → prestige loop
- Deity influence evolves based on history, worship, failures

### Community / stream friendliness
- Systems that create compelling emergent stories
- Optional audience-facing “witness logs” or event recaps

## Scope boundaries (current)
- Single-player first; multiplayer/co-op only if it does not compromise determinism or narrative clarity.
- Avoid over-automating: the settlement should feel alive, not “solved”.

## Deliverable expectation
This spec aims to be **implementation-oriented**: each module includes mechanics, data needs, tuning knobs, and UI considerations.
